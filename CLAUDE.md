# Monthly Giving Platform ŌĆō Claude Context

## Project Overview

A subscription-based alumni giving platform for a school. Alumni contribute monthly based on how many years they have been an alum.

**Core Business Rule:**
```
monthly_amount = (current_year - graduation_year) ├Ś $1
```

Implementation: Stripe price = $1/month, subscription quantity = years_out. Quantity is updated automatically before each billing cycle via webhooks ŌĆö no cron jobs.

## Architecture

### Layer Rules (strict separation)
- **Route handlers** (`app/api/`) ŌĆö HTTP only. No business logic.
- **Domain** (`src/domain/`) ŌĆö Pure business logic. No HTTP, no DB.
- **Services** (`src/services/`) ŌĆö Stripe/Auth orchestration.
- **Repos** (`src/repos/`) ŌĆö DB access only. All Drizzle queries live here.
- **lib** (`src/lib/`) ŌĆö Shared utilities, clients.

The browser never talks directly to the database.

### Folder Structure
```
app/
  api/
    billing/          # checkout-session, portal-session
    webhooks/stripe/  # Stripe webhook handler
    me/               # authenticated user profile
    admin/            # admin-only endpoints
```

## Billing Flow (Webhook-Driven, No Cron)

1. User subscribes via Stripe Checkout
2. Stripe creates subscription; webhook stores mapping in `billingSubscriptions`
3. Before renewal, Stripe sends `invoice.upcoming`
4. API calculates new quantity (years_out), updates Stripe subscription (no proration)
5. Stripe generates invoice and charges the correct amount

Webhook events handled: `invoice.upcoming`, `invoice.created`

Webhook handling is idempotent ŌĆö check `stripeEvents` before processing.

## Local Development

```bash
# Start Docker DBs (dev on :5432, test on :5433, Adminer UI on :8080)
npm run db:up

# Apply migrations to dev DB
npm run drizzle:migrate

# Apply migrations to test DB
npm run drizzle:migrate:test

# Reset dev DB
npm run db:reset

# Reset test DB
npm run db:reset:test

# Start Next.js dev server
npm run dev
```

Environment variables are in `.env.local` (see `.env.example`).

### Schema Change Workflow

When modifying `src/db/schema.ts`, always follow this sequence ŌĆö never skip steps:

```bash
# 1. Edit src/db/schema.ts
# 2. Generate a migration file
npm run drizzle:generate

# 3. Apply to dev DB
npm run drizzle:migrate

# 4. Apply to test DB
npm run drizzle:migrate:test
```

Commit both the schema change and the generated migration file together.

### Stripe CLI ŌĆö Local Webhook Testing

Stripe webhooks can't reach localhost without the Stripe CLI. To test webhooks locally:

```bash
stripe listen --forward-to localhost:3000/api/webhooks/stripe
```

This proxies real Stripe events to your local server and prints the webhook signing secret to use in `.env.local`.

## Development Rules for Claude

### Modifying This File
Never edit `CLAUDE.md` autonomously. Always show the proposed change and get explicit confirmation from the developer before modifying this file, regardless of permission mode or how you were invoked.

### After Every Code Change

1. **Lint** ŌĆö Run `npm run lint` and fix all errors before considering work done.
2. **Format** ŌĆö Run `npm run format` to apply Prettier.
3. **Tests** ŌĆö Run `npx vitest run` if test files exist. Note: vitest is installed but tests are not yet set up. When writing tests, use the test DB (`DATABASE_URL_TEST`), never the dev DB.
4. **Tests for new features** ŌĆö New features should have corresponding tests. If the test infrastructure isn't set up yet, note what should be tested so it can be added later.
5. **README** ŌĆö Update `README.md` if the change affects setup, configuration, environment variables, or developer workflow. Do not update it for every feature.
6. **CLAUDE.md** ŌĆö If a change affects the schema, architecture, scripts, or established patterns documented here, propose an update to `CLAUDE.md` and wait for confirmation before applying it.

### Code Style
- **Verbose naming** ŌĆö prefer `error` over `e`, `subscription` over `sub`, `user` over `u`. Exception: established conventions like `i` in loops.
- **Clarity over cleverness** ŌĆö write code that is easy to read and reason about, not code that is impressively terse.
- **Avoid duplication** ŌĆö extract shared logic rather than copying it. Three similar lines may be fine; a fourth means it needs a name.
- **No `any`** ŌĆö TypeScript strict mode is enabled. Use `unknown` and narrow it if the shape is dynamic. Never use `any` as an escape hatch.
- **No magic values** ŌĆö extract repeated string literals and numbers to named constants.
- **Explicit error handling** ŌĆö never silently swallow errors in `catch` blocks. Always log, propagate, or handle them intentionally.

### Import Paths
Prefer `@/` for cross-directory imports ŌĆö `@/*` maps to the project root (configured in `tsconfig.json`). For example, use `@/src/repos/user` rather than `../../../src/repos/user`. Same-directory relative imports (e.g., `./schema`) are fine.

### Layer Separation (Mandatory)
- Route handlers contain only HTTP parsing and response ŌĆö no business logic
- Business logic lives in `src/domain/` only
- All DB queries go through `src/repos/` ŌĆö never call `db` directly from route handlers or domain
- Services in `src/services/` orchestrate between domain and external APIs (Stripe, Supabase)

### Database Queries
Write sargable WHERE clauses ŌĆö conditions that allow Postgres to use indexes. Avoid applying functions or casts to indexed columns on the left-hand side of a condition.

```ts
// Bad ŌĆö non-sargable, can't use index on created_at
.where(sql`YEAR(${table.createdAt}) = 2024`)

// Good ŌĆö sargable, index on created_at can be used
.where(and(gte(table.createdAt, new Date('2024-01-01')), lt(table.createdAt, new Date('2025-01-01'))))
```

### External Documentation
Stripe, Drizzle ORM, Next.js App Router, and Supabase Auth all evolve rapidly. When implementing anything non-trivial with these libraries, fetch the current official docs rather than relying on training data. Prefer official docs over blog posts or Stack Overflow.

## Guardrails ŌĆö What NOT to Do

- Do not calculate billing or update quantities via cron jobs
- Do not allow the frontend to modify billing tables
- Do not enable proration when adjusting subscription quantity
- Do not duplicate Stripe billing logic in the database
- Do not put domain logic inside route handlers
- Do not call the database directly from route handlers ŌĆö use repos
- Do not run tests against the dev database (`DATABASE_URL`) ŌĆö use `DATABASE_URL_TEST`
- Do not use `npm run drizzle:push` ŌĆö it bypasses the migration system entirely. Always use `drizzle:generate` + `drizzle:migrate` to keep migrations in sync
